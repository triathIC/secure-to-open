# S3E2 – Dein Playbook ist kein Prozess

## Executive Summary

Die meisten Organisationen verwechseln Dokumentation mit Prozess. Ein Playbook in Confluence beschreibt, was passieren soll. Ein Prozess stellt sicher, dass es passiert – unabhängig davon, wer gerade Schicht hat. Der Unterschied ist nicht semantisch, sondern architektonisch: Ein Prozess hat definierte Inputs, Entscheidungslogik, technische Schnittstellen und messbare Outputs.

## Das Problem: Playbooks als Sicherheitstheater

### Anatomie eines typischen Playbooks

Die meisten Playbooks folgen demselben Muster:

1. Jemand schreibt nach einem Incident auf, was man hätte tun sollen
2. Das Dokument wird in ein Wiki gestellt
3. Niemand testet, ob es funktioniert
4. Beim nächsten Incident liest es niemand, weil keine Zeit ist

Das Ergebnis: Das Playbook existiert für das Audit, nicht für den Betrieb.

### Was einem Playbook fehlt

| Element | Playbook (typisch) | Prozess (architektonisch) |
|---------|-------------------|--------------------------|
| Trigger | "Wenn ein Alert kommt..." | Automation Rule mit definierten Kriterien |
| Input | "Prüfe die Logs..." | Enrichment-Query liefert Kontext automatisch |
| Entscheidung | "Bewerte die Severity..." | Decision Tree mit definierten Schwellwerten |
| Aktion | "Deaktiviere den Account..." | API-Call an Entra ID mit Rollback-Option |
| Output | "Dokumentiere im Ticket..." | Structured Data im Incident-Objekt |
| Ownership | "Das SOC-Team..." | Benannter Process Owner mit Eskalationspfad |

## Von Prosa zu Architektur

### Sentinel Automation Rules als Prozess-Engine

```kql
// Beispiel: Automatische Incident-Enrichment statt "Prüfe die Logs"
// Dies ersetzt Schritt 2-3 eines typischen Playbooks
let IncidentEntities = SecurityIncident
| where TimeGenerated > ago(1h)
| where Status == "New"
| mv-expand parse_json(AlertIds)
| join kind=inner (
    SecurityAlert
    | mv-expand parse_json(Entities)
    | extend EntityType = tostring(Entities.Type),
             EntityValue = coalesce(
                 tostring(Entities.AadUserId),
                 tostring(Entities.HostName),
                 tostring(Entities.Address)
             )
) on $left.AlertIds == $right.SystemAlertId;
//
// Automatischer Kontext statt manueller Recherche
IncidentEntities
| where EntityType == "account"
| join kind=leftouter (
    IdentityInfo
    | summarize arg_max(TimeGenerated, *) by AccountUPN
) on $left.EntityValue == $right.AccountObjectId
| extend AutoEnrichment = bag_pack(
    "Department", Department,
    "JobTitle", JobTitle,
    "RiskLevel", RiskLevel,
    "IsPrivileged", isnotempty(AssignedRoles),
    "Manager", Manager
)
| project IncidentNumber, EntityValue, AutoEnrichment
```

## Prozess-Reifegrad-Modell

| Stufe | Beschreibung | Playbook-Charakter | Testbar? |
|-------|-------------|-------------------|----------|
| 0 – Tribal Knowledge | "Frag Markus, der weiß das" | Keins | Nein |
| 1 – Dokumentiert | Confluence-Seite mit Schritten | Prosa | Nein |
| 2 – Strukturiert | Definierte Inputs/Outputs, Entscheidungsbaum | Flowchart | Manuell |
| 3 – Teilautomatisiert | Logic App für Enrichment + Routing | Code + Prosa | Teilweise |
| 4 – Vollautomatisiert | End-to-End mit Rollback und Feedback | Code | Ja, kontinuierlich |

Die meisten Organisationen glauben, auf Stufe 2-3 zu sein. Die Realität ist meist Stufe 0-1.

## Key Takeaways

1. Ein Playbook, das niemand unter Druck liest, ist kein Sicherheitsmechanismus.
2. Prozess-Qualität zeigt sich nicht an der Dokumentation, sondern an der Testbarkeit.
3. Jeder manuelle Schritt in einem Playbook ist ein potenzieller Single Point of Failure.
4. Der erste Schritt von Playbook zu Prozess ist nicht Automation – sondern die Definition von Inputs, Outputs und Entscheidungskriterien.
